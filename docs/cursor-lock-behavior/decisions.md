# 智能光标锁定行为 — 决策日志

按时间倒序记录本次修复过程中的关键决策。

## D-003: 锁定后通过显式调用恢复，而非自动超时

**日期**：2026-07-05
**状态**：已实现

### 背景

锁定行为实现后，需要决定如何让光标"解锁"恢复鼠标跟随。

### 选项

A) **自动超时解锁**：`isStuck = true` 后 N 毫秒自动清除
B) **显式调用解锁**：调用 `sdk.releaseCursor()` 主动释放
C) **跟随 hover 离开解锁**：当鼠标离开目标时解锁

### 决定

选择 **B) 显式调用解锁**。

### 理由

1. **可预测性**：调用方完全掌握解锁时机，避免隐式状态切换导致 bug
2. **灵活性**：不同场景可以用不同的解锁策略（旅程结束、用户点击、超时等）
3. **简单**：实现成本最低，不需要管理 timer
4. **未观测到需要自动超时**：当前没有用户场景需要"锁一段时间后自动恢复"

### 影响

调用方需要主动管理解锁逻辑。当前调用方都还在锁定语义下工作正常：
- 单次查询：锁定直到下一次查询或 deactivate
- Journey：每个 step 重新 flyTo 自动清除锁定

## D-002: 动画完成后立即锁定，不设延迟

**日期**：2026-07-05
**状态**：已实现

### 背景

实现锁定时需要决定：动画完成后立即锁定还是延迟 N 毫秒锁定？

### 选项

A) **立即锁定**：动画结束的下一帧就把 `isStuck` 设为 true
B) **延迟锁定**：动画结束后等待 100-500ms 才锁定

### 决定

选择 **A) 立即锁定**。

### 理由

1. **简洁**：状态机只有"动画中"和"锁定"两个状态，没有中间过渡态
2. **避免边界 bug**：延迟窗口内如果发生鼠标移动，光标会跳走，破坏锁定语义
3. **实测反馈**：用户希望"一直停留在那"，延迟会让光标在动画最后一刻突然跳一下

### 影响

光标在动画最后一帧到锁定之间没有空隙，鼠标移动被立即屏蔽。实测过程中光标位置稳定保持在目标坐标。

## D-001: 用两个独立标志 isAnimating / isStuck 建模状态

**日期**：2026-07-05
**状态**：已实现

### 背景

需要让 `handleMouseMove` 在动画期间和锁定期间都忽略鼠标移动。

### 选项

A) **单一标志 `locked`**：动画时为 true，锁定后保持 true，靠 try/finally 管理
B) **两个独立标志** `isAnimating` 和 `isStuck`：分别表示不同语义
C) **一个枚举状态** `CursorState = 'idle' | 'flying' | 'stuck'`

### 决定

选择 **B) 两个独立标志**。

### 理由

1. **关注点分离**：`isAnimating` 由动画循环管，`isStuck` 由 flyTo 的 finally 管
2. **类型简单**：boolean 比 enum 更轻，不需要 switch
3. **组合表达**：未来如果需要"动画中但允许鼠标取消"等组合，只需要修改一个标志

### 未来演进

如果出现更复杂的状态转换（例如 cancel、queue、retry），再考虑迁移到 enum 方案（选项 C）。当前需求下选项 B 已经足够清晰。
